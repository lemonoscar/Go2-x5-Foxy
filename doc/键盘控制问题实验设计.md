# 键盘控制问题实验设计

## 1. 背景

当前 `rl_ras_n` 中的键盘控制链路不仅负责人工输入，还承担了核心状态推进职责。现状表现为：

- 键盘输入会直接驱动 `Passive -> GetUp -> RLLocomotion` 状态切换。
- 启动时序会通过注入 `Num0`、`Num1` 的方式复用键盘链路。
- 部分保护逻辑也会直接写入键盘状态。
- 输入事件当前通过单个原子槽传递，而不是事件队列。

这意味着键盘已经不是外围输入设备，而是系统内部控制总线的一部分。只要这条链路存在事件丢失、时序错位、状态粘滞或线程阻塞问题，就会直接影响主流程稳定性。

本实验设计的目标不是先证明某一个“死锁点”，而是通过分阶段实验，定位当前键盘控制方案的真实问题边界，并验证一个可替换的改造方案。

## 2. 实验目标

本实验设计围绕以下四个目标展开：

1. 明确当前键盘控制链路的具体问题，而不是停留在“可能死锁”的描述。
2. 验证当前键盘事件传递是否存在覆盖、丢失、粘滞、时序错位问题。
3. 验证将“键盘键值驱动”替换为“显式操作命令驱动”后，能否消除当前主要风险。
4. 为后续时间控制、远程控制、自动启动控制提供统一输入架构。

## 3. 当前问题假设

基于现有代码，先建立以下实验假设。

### 3.1 假设 A：输入通道存在事件覆盖问题

当前键盘输入通过单个 `pending_keyboard_input` 原子变量传递，而不是队列。因此当多个事件在一个控制周期内连续到来时，后一个事件可能覆盖前一个事件，导致事件丢失。

### 3.2 假设 B：键盘语义与业务语义混杂

当前系统使用 `Num0`、`Num1`、`P`、`R` 等键值直接表示业务动作，这会导致：

- 键盘输入和自动启动、保护逻辑复用同一通道。
- 业务逻辑不得不理解键盘键值，而不是理解“请求起身”“请求进入 RL”等显式意图。

### 3.3 假设 C：FSM 依赖键状态而非请求事件

当前 FSM 直接读取 `current_keyboard`，本质上依赖“当前一拍键值是什么”，而不是“是否收到过待处理请求”。这会导致状态切换窗口与输入时序强耦合。

### 3.4 假设 D：`Num1` 存在粘滞语义污染

当前 `Num1` 同时承担“进入 RL 模式”和“固定命令锁存”的职责。这会导致一个按键承担多种含义，增加链路不透明性，也增加误触发和状态残留风险。

## 4. 实验总体策略

实验分三阶段进行：

1. 现状诊断实验  
   不改架构，只增加观测，确认问题是否真实存在。

2. 输入抽象实验  
   引入统一操作命令层，但尽量不大改 FSM，验证输入链路是否先变稳定。

3. 状态机解耦实验  
   将 FSM 从键值驱动改为请求驱动，验证整体链路是否可稳定替代现有方案。

## 5. 阶段一：现状诊断实验

### 5.1 目标

验证当前键盘控制链路在真实运行时是否存在如下问题：

- 事件覆盖
- 事件丢失
- 键值粘滞
- 状态切换时序依赖
- 自动启动与人工输入互相干扰

### 5.2 方法

在不改变现有业务行为的前提下，增加以下诊断日志和统计项：

- 键盘采集时间戳
- 原始字节值
- 映射后的键值
- `pending_keyboard_input` 写入计数
- `pending_keyboard_input` 覆盖计数
- 控制线程消费时间戳
- FSM 当前状态
- 本周期处理的状态切换结果
- 自动启动注入来源标识
- 保护逻辑注入来源标识

### 5.3 观测点

建议增加以下观测点：

- `KeyboardInterface()` 采集事件时记录一次。
- `StateController()` 读取待处理输入时记录一次。
- `fsm.Run()` 前后记录状态名。
- `MaybeRunStartupSequence()` 注入 `Num0/Num1` 时记录来源。
- `AttitudeProtect()` 写入保护动作时记录来源。

### 5.4 关键指标

- 单位时间内采集事件数
- 单位时间内消费事件数
- 覆盖事件数
- 未消费事件数
- 状态切换成功率
- 进入 `GetUp` 后成功进入 `RLLocomotion` 的比例
- 同一键值连续保留周期数

### 5.5 预期结果

如果阶段一假设成立，应至少观察到以下一种或多种现象：

- 短时间连续输入时，部分事件只记录采集而未记录消费。
- 自动启动注入与人工输入在同一控制周期内发生覆盖。
- `Num1` 在多个周期中持续存在，而不是单次请求。
- 某些状态切换高度依赖输入恰好落在特定时间窗口。

### 5.6 验收结论

阶段一结束后，应得到一份基于日志和统计的明确结论：

- 当前键盘链路是否存在覆盖问题
- 当前键盘链路是否存在粘滞问题
- 当前 FSM 是否高度依赖输入时序窗口
- 当前链路是否适合作为统一控制入口继续沿用

## 6. 阶段二：输入抽象实验

### 6.1 目标

在尽量少改行为的前提下，引入统一“操作命令层”，验证是否可以先将键盘从业务语义中剥离。

### 6.2 核心设计

新增显式命令类型 `OperatorCommand`，而不是继续直接传播 `Keyboard` 枚举。

建议命令集合如下：

- `RequestGetUp`
- `RequestEnterRl`
- `RequestGetDown`
- `RequestManualArmHold`
- `RequestManualArmHome`
- `RequestZeroVelocity`
- `RequestEStop`
- `RequestFaultReset`

同时为每条命令记录：

- 来源
- 时间戳
- 序号

来源建议枚举如下：

- `Keyboard`
- `StartupSequence`
- `SafetyProtection`
- `Gamepad`
- `FutureRemote`

### 6.3 实验改造范围

本阶段只替换输入传递方式，不立即重写 FSM。

具体做法：

- 键盘采集层把键值映射为 `OperatorCommand`
- 启动时序不再写 `Num0/Num1`，改为发 `RequestGetUp` / `RequestEnterRl`
- 姿态保护不再写 `P`，改为发显式保护命令
- 用小型有界队列替换原子单槽

### 6.4 队列要求

输入命令通道至少满足以下要求：

- 单生产者/多生产者都可接入
- 控制线程为唯一消费者
- 保证 FIFO
- 队列满时有明确丢弃策略和日志
- 可统计丢包数、消费数、积压长度

### 6.5 关键验证点

- 连续快速输入不再覆盖前序命令
- 不同来源的命令不会互相污染
- 输入日志不再依赖键值语义
- 启动时序与键盘输入能够共存

### 6.6 预期结果

如果阶段二成功，应看到：

- 所有输入都通过统一命令通道进入系统
- 键盘只负责产生命令，而不再承载业务语义
- 启动时序和保护逻辑不再伪装成按键
- 系统已经具备从键盘迁移到其他输入源的基础

## 7. 阶段三：状态机解耦实验

### 7.1 目标

将 FSM 从“键状态驱动”改造为“请求驱动”，彻底消除当前状态切换与键值的耦合。

### 7.2 核心设计

引入操作请求状态，而不是直接读取 `current_keyboard`。

建议维护如下待处理标志：

- `pending_get_up`
- `pending_enter_rl`
- `pending_get_down`
- `pending_manual_arm_hold`
- `pending_manual_arm_home`
- `pending_zero_velocity`
- `pending_estop`
- `pending_fault_reset`

请求进入系统后不要求“本周期刚好匹配状态窗口”，而是按以下原则处理：

- 请求先进入待处理区
- FSM 按状态和条件消费请求
- 消费成功后清除请求
- 若当前状态暂不满足消费条件，则请求保留

### 7.3 状态切换原则

建议按如下方式重写状态切换逻辑：

- `Passive` 收到 `pending_get_up` 后进入 `GetUp`
- `GetUp` 完成后，如果存在 `pending_enter_rl`，自动切入 `RLLocomotion`
- `RLLocomotion` 收到 `pending_get_down` 后进入 `GetDown`
- `EStop`、`FaultReset` 采用更高优先级逻辑
- `ManualArm` 请求走独立控制支路，不再依赖键值残留

### 7.4 关键验证点

- `RequestEnterRl` 即使早于 `GetUp` 完成到达，也不会丢失
- `GetUp` 完成后可以自动消费挂起的 RL 请求
- FSM 决策逻辑不再读取 `Keyboard::Num0/Num1/P`
- 同一个请求不会被重复消费

### 7.5 预期结果

如果阶段三成功，应实现以下效果：

- 键盘不再参与状态机内部语义表达
- 自动启动流程不再依赖模拟按键
- 时间控制和键盘控制可以统一接入
- 未来即使完全移除键盘，也不会影响主流程架构

## 9. 测试设计

### 9.1 单元测试

建议新增或补充以下单测：

- 多次连续命令入队后顺序消费正确
- 队列满时丢弃策略正确且有计数
- `RequestEnterRl` 先到、`GetUp` 后完成时可正确挂起并消费
- `RequestEStop` 具有高优先级
- `Num1` 不再承担粘滞状态职责

### 9.2 集成测试

建议增加以下集成测试场景：

1. 键盘单独控制  
   验证 `GetUp -> RL -> GetDown` 正常。

2. 启动时序单独控制  
   验证不接键盘时也能正常推进流程。

3. 键盘与启动时序并存  
   验证二者不会互相覆盖或污染。

4. 安全保护打断场景  
   验证保护命令不会依赖键值注入。

5. 高频输入场景  
   验证连续操作不会导致事件丢失。

### 9.3 台架验证

台架验证建议重点观察：

- 输入日志顺序
- 状态机切换顺序
- `GetUp` 完成后进入 RL 的稳定性
- 保护动作触发后的恢复行为
- 无交互终端时系统行为

## 10. 验收标准

满足以下条件可认为第一个问题已基本解决：

- 键盘不再承担系统内部控制总线职责
- 所有来源统一进入操作命令通道
- 不再使用单槽原子变量传递关键输入
- FSM 不再直接依赖键值判断主状态切换
- 自动启动流程不再通过模拟按键实现
- 连续输入和异步输入场景下无明显事件丢失
- 至少存在一组覆盖“挂起请求后延迟消费”的自动化测试

## 11. 风险与注意事项

- 该改造会影响 `go2_x5` 的输入语义，需要防止破坏现有人工操作习惯。
- `Num1` 相关逻辑目前同时耦合固定速度命令，需要在改造中单独拆分，不可直接删除。
- 如果仅替换输入通道而不改 FSM，问题只能部分缓解，不能根除。
- 若台架验证不足，可能出现“仿真通过、真机时序仍出问题”的情况。

## 12. 结论

针对第一个问题，推荐的解决方向不是继续修补键盘线程，而是将系统从“键值驱动”重构为“显式操作命令驱动”。

实验应先诊断现状，再替换输入通道，最后重构 FSM。只有在三阶段都完成后，键盘控制问题才算真正被解决，而不是仅仅被掩盖。

## 13. 第一轮诊断结果（2026-04-16）

本轮未修改运行行为，先基于代码路径对第一阶段进行了静态诊断。结论是：当前键盘控制问题的具体来源已经可以明确定位，不需要继续停留在“可能死锁”的模糊表述。

### 13.1 来源一：输入传递层使用单槽原子变量，存在覆盖风险

当前键盘输入不是通过事件队列传递，而是通过单个原子变量 `pending_keyboard_input` 传递。

对应代码：

- `src/rl_sar/library/core/rl_sdk/rl_sdk.cpp:81-85`
- `src/rl_sar/library/core/rl_sdk/rl_sdk.cpp:765`

具体表现：

- 生产者侧使用 `store(...)`
- 消费者侧使用 `exchange(...)`
- 该结构只能保存“最后一个事件”，不能保存一段时间内的多个事件

因此只要多个事件在一次控制消费前连续到来，前面的事件就可能被后面的事件覆盖。这个问题属于输入传递层的结构性缺陷。

### 13.2 来源二：键盘通道被多个业务路径复用，语义已经被污染

当前系统不只有真实键盘线程会写入键盘状态，其他控制逻辑也会直接调用 `SetKeyboard(...)`。

已确认的来源包括：

- 人工键盘采集  
  `src/rl_sar/library/core/rl_sdk/rl_sdk.cpp:687-765`
- 自动启动时序  
  `src/rl_sar/src/rl_sar/core/rl_real_go2_x5.cpp:2927-2968`
- 姿态保护逻辑  
  `src/rl_sar/library/core/rl_sdk/rl_sdk.cpp:512-529`

这说明键盘通道当前同时承载：

- 人工输入
- 自动启动控制
- 保护控制

因此问题已经不是“键盘线程有没有 bug”，而是“系统把不同来源的控制意图都塞进了键盘键值语义里”。

### 13.3 来源三：状态机直接读取当前键值，而不是读取显式请求

FSM 的状态切换直接依赖 `rl.control.current_keyboard`。

对应代码：

- `src/rl_sar/fsm_robot/fsm_go2_x5.hpp:96-110`

例如 `GetUp -> RLLocomotion` 的切换并不是“收到进入 RL 请求后等待条件满足”，而是“只有当 `percent_getup >= 1.0f` 的那一拍，`current_keyboard == Num1` 时才切换”。

这意味着：

- FSM 依赖瞬时键值
- FSM 依赖时机窗口
- 输入事件没有挂起机制

因此状态机消费层本身也具有结构性脆弱性。

### 13.4 来源四：`Control` 结构采用“当前键值 + 上一键值”模型，导致状态粘滞

`Control` 结构并不是纯事件模型，而是“当前键值/上一键值”模型。

对应代码：

- `src/rl_sar/library/core/rl_sdk/rl_sdk.hpp:100-130`

其中：

- `SetKeyboard(...)` 会修改 `current_keyboard` 与 `last_keyboard`
- `ClearInput()` 不是清空为 `None`，而是把 `current_keyboard` 回退为 `last_keyboard`

这意味着键盘语义不是单次事件，而更像一个带残留的状态模型。该设计会天然放大粘滞问题。

### 13.5 来源五：`Num1` 被赋予双重语义，产生额外耦合

`Num1` 既被用作“请求进入 RL 状态”，又被用作“固定速度命令锁存”的触发键。

对应代码：

- `src/rl_sar/library/core/rl_sdk/rl_sdk.cpp:133-149`
- `src/rl_sar/src/rl_sar/go2x5/control/go2_x5_operator_control.cpp:39-45`
- `src/rl_sar/src/rl_sar/core/rl_real_go2_x5.cpp:3008-3014`

并且在处理 `Key[1] pressed` 后，代码会显式执行：

- `this->control.last_keyboard = Input::Keyboard::Num1;`

这使得 `Num1` 不再只是一次性事件，而被人为变成了带保持语义的状态标记。该设计会干扰后续输入判断，也会增加状态残留风险。

### 13.6 来源六：主控制循环把键值同时用于 FSM 和 supervisor/operator 控制

主循环中，在 `MaybeRunStartupSequence()` 和 `StateController()` 之后，又继续根据 `current_keyboard` 更新：

- `operator_estop_requested_`
- `operator_fault_reset_requested_`
- `operator_manual_arm_requested_`
- 速度清零逻辑

对应代码：

- `src/rl_sar/src/rl_sar/core/rl_real_go2_x5.cpp:2973-3055`

这说明同一个键值在同一周期内同时被：

- FSM 消费
- supervisor 输入消费
- operator 控制消费

因此同一份输入状态承担了多路业务语义，属于典型的高耦合输入总线设计。

### 13.7 当前阶段结论

第一轮诊断后，可以明确认定：

1. 当前问题的首要来源不是单一“死锁点”，而是输入链路设计本身。
2. 具体来源位于三层：
   - 输入传递层：单槽原子变量，存在覆盖风险
   - 输入语义层：多个来源复用键盘语义，存在污染问题
   - 状态机消费层：直接依赖键值与时间窗口，存在脆弱性
3. 即使后续没有发现死锁，当前键盘控制方案也仍然需要重构。

### 13.8 下一步实验重点

下一步建议进入“带观测的最小改造实验”，优先验证两件事：

1. 将单槽原子变量替换为事件队列后，输入覆盖问题是否消失。
2. 将启动时序和保护逻辑从 `SetKeyboard(...)` 改为显式命令后，状态切换行为是否更稳定。

## 14. 第二轮实验结果（2026-04-16）

本轮按实验设计执行了“输入传递层最小替换实验”，目标是先验证输入覆盖问题是否确实来自单槽原子变量，而不是来自 FSM 或 operator 逻辑。

### 14.1 实验内容

本轮只修改输入传递层，不修改上层状态机语义。

具体改动如下：

- 将 `pending_keyboard_input` 单槽原子变量替换为 FIFO 队列
- 新增：
  - `EnqueueKeyboardInput(...)`
  - `ConsumePendingKeyboardInput()`
  - `PendingKeyboardInputCount()`
- `StateController()` 改为每周期消费一个排队事件
- `KeyboardInterface()` 改为将输入事件入队
- 保持现有 FSM、`Control`、`MaybeRunStartupSequence()` 行为不变

### 14.2 实验目的

本轮实验只回答一个问题：

当前键盘输入的覆盖问题，是否确实来自“单槽原子变量只能保存最后一个事件”。

### 14.3 新增验证

新增测试：

- `test_go2_x5_keyboard_input_queue`

该测试验证：

- 多个键盘输入按顺序入队
- 多个键盘输入按顺序消费
- `None` 事件不会污染队列

同时保留并更新：

- `test_go2_x5_keyboard_input_contract`

用于确认原有输入链路的关键行为约束仍然成立。

### 14.4 实验结果

定向测试结果如下：

- `test_go2_x5_keyboard_input_contract` 通过
- `test_go2_x5_keyboard_input_queue` 通过

这说明：

1. 输入层已经从“只保留最后一个事件”变为“按顺序保留多个事件”
2. 多个输入在一次控制消费前到达时，不再被单槽覆盖
3. 输入覆盖问题至少在传递层已被局部消除

### 14.5 当前可确认结论

经过第二轮实验，可以确认：

- 第一轮诊断中的“来源一：输入传递层单槽原子变量”判断是正确的
- 该问题可以通过 FIFO 队列的最小替换稳定复现并修正
- 当前输入链路的一个核心不稳定来源已经被明确定位并局部验证

### 14.6 本轮实验的边界

本轮实验没有解决以下问题：

- 启动时序仍然直接调用 `SetKeyboard(...)`
- 姿态保护仍然直接调用 `SetKeyboard(...)`
- FSM 仍然直接读取 `current_keyboard`
- `Num1` 的双重语义和粘滞问题仍然存在
- `Control` 的 `current_keyboard / last_keyboard` 状态模型仍然存在

因此，本轮只能说明：

- “输入覆盖”这个来源已经被验证，并已有有效修复方向

但还不能说明：

- 键盘控制整体问题已经解决
- 状态切换时序问题已经解决
- 自动启动与保护逻辑的语义污染已经解决

### 14.7 第二轮实验结论

第二轮实验支持如下结论：

> 当前键盘控制问题的第一类具体来源，确实来自输入传递层的单槽设计。

该问题已经被最小改造实验验证，并证明 FIFO 队列是有效修复方向。

### 14.8 下一步实验建议

下一步应继续按设计推进，优先做第三轮最小实验：

1. 将 `MaybeRunStartupSequence()` 从 `SetKeyboard(Num0/Num1)` 改为显式命令请求
2. 将保护逻辑从 `SetKeyboard(P)` 改为显式控制请求
3. 在不完全重写 FSM 的前提下，先验证“请求挂起”机制能否消除 `GetUp -> RL` 的时序窗口问题

## 15. 第三轮实验结果（2026-04-16）

本轮实验聚焦一个最核心的用户行为：

> 按下 `0` 后，机器人必须能够进入 `GetUp`。

本轮不追求一次性解决所有键盘控制问题，而是优先验证“起身请求”是否可以从瞬时键值中解耦出来，变成可挂起、可消费的显式请求。

### 15.1 实验内容

本轮做了以下最小改造：

- 在 `RL` 中新增显式请求：
  - `RequestGetUp()` / `ConsumeGetUpRequest()`
  - `RequestEnterRl()` / `ConsumeEnterRlRequest()`
  - `RequestGetDown()` / `ConsumeGetDownRequest()`
  - `RequestPassive()` / `ConsumePassiveRequest()`
- `StateController()` 中将当前输入映射为显式请求，而不是仅依赖瞬时键值
- `go2_x5` 的 FSM 改为优先消费这些显式请求，而不是直接判断 `current_keyboard`
- `MaybeRunStartupSequence()` 改为发出显式请求，而不再直接写入 `Num0/Num1`
- `AttitudeProtect()` 改为发出显式 passive 请求，而不再直接写 `P`

### 15.2 核心验证点

重点验证：

- 键盘 `0` 进入系统后，即使 `current_keyboard` 后续被清除，请求仍会保留到 FSM 消费为止
- `Passive -> GetUp` 的状态推进不再依赖“某一拍刚好看到 `Num0`”

### 15.3 新增测试

新增测试：

- `test_go2_x5_getup_request`

该测试显式验证：

1. 向输入系统注入 `Num0`
2. 第一周期只完成“请求挂起 + FSM 记录状态切换意图”
3. 即使随后调用 `ClearInput()` 清掉当前输入态
4. 下一周期仍能稳定进入 `RLFSMStateGetUp`

### 15.4 测试结果

本轮定向测试结果如下：

- `test_go2_x5_keyboard_input_contract` 通过
- `test_go2_x5_keyboard_input_queue` 通过
- `test_go2_x5_getup_request` 通过

这说明：

- 输入 FIFO 队列替换没有破坏现有输入契约
- `0 -> GetUp` 的核心行为已经由显式请求机制稳定支撑
- 起身行为已不再依赖瞬时键值窗口

### 15.5 当前可确认结论

经过第三轮实验，可以明确确认：

1. “按下 0 后机器人能做到 getup” 这一核心链路已经被最小改造修正
2. 该链路当前的可靠性来自“请求挂起并由 FSM 消费”，而不是来自“当前键值刚好命中”
3. 第一阶段定位出的“FSM 直接读取瞬时键值”问题，已经在 `go2_x5` 起身链路上被实质性削弱

### 15.6 当前仍未完全解决的问题

虽然 `0 -> GetUp` 已经完成修正，但以下问题仍未完全关闭：

- `GetUp -> RL` 的后续链路还需要继续验证请求挂起是否完全覆盖时序窗口
- `Control` 的 `current_keyboard / last_keyboard` 模型仍然存在
- 其他机器人类型的 FSM 仍然还是旧的键值驱动模型
- supervisor/operator 层还存在部分历史键值耦合

### 15.7 本轮结论

第三轮实验已经完成了当前最重要的核心目标：

> 现在 `go2_x5` 这条链路上，按下 `0` 后，系统已经具备稳定进入 `GetUp` 的机制。

这不是经验判断，而是已经由自动化测试直接覆盖并验证通过。

## 16. 第四轮实验结果（2026-04-16）

本轮实验继续围绕第二个核心行为推进：

> 按下 `1` 后，系统必须能够稳定进入 RL，而不是因为时序窗口问题卡在 `GetUp`。

### 16.1 实验目标

本轮不追求完整验证 RL 模型加载、推理或真机控制效果，而是先验证更底层、更关键的一件事：

- `EnterRl` 请求是否能在 `GetUp` 未完成时挂起
- `GetUp` 完成后，该请求是否仍能被正确消费为 `RLLocomotion` 切换意图

### 16.2 实验方法

本轮沿用第三轮中引入的显式请求机制，不再依赖瞬时 `current_keyboard == Num1`。

验证重点放在：

- `RequestEnterRl()` 发出后不立即丢失
- `RLFSMStateGetUp::CheckChange()` 在 `percent_getup < 1.0f` 时不会错误消费
- `percent_getup >= 1.0f` 后，挂起的 `EnterRl` 请求能够被消费

3. 连续执行 `Run()` 直到 `GetUp` 插值完成
4. 检查 `CheckChange()` 是否返回 `RLFSMStateRLLocomotion`
5. 确认请求已被消费而不是无限残留

### 16.4 测试结果

### 16.6 重要边界说明

也就是说，目前已经验证：

但还没有在这个实验里验证：

更准确地说：

- `1` 不再是“必须命中某一拍的瞬时键值”
- 而是“可挂起并在条件满足后消费的显式进入 RL 请求”

## 17. 第五轮实验结果（2026-04-16）

本轮实验完成最后一段闭环验证，重点回答下面这个问题：

> 状态切进 `RLLocomotion` 之后，会不会因为 `Enter()` 初始化失败而立刻掉回 `Passive`，或者在成功时能够稳定留在 RL 状态。

### 17.1 实验目标

前四轮实验已经验证了：

- `0 -> GetUp` 的请求机制稳定
- `1` 的进入 RL 请求可以挂起并在 `GetUp` 完成后消费

但还缺少最后一个关键环节：

- `RLLocomotion::Enter()` 成功时，系统是否真正稳定留在 RL
- `RLLocomotion::Enter()` 失败时，系统是否按预期回退，而不是进入不确定状态

### 17.2 最小可测改造

为了让这一环能在自动化测试中可控验证，本轮做了一处最小改造：

- 将 `RL::InitRL(...)` 改为可覆写接口

这样在测试中就可以精确模拟两类情况：

1. `InitRL()` 成功
2. `InitRL()` 抛异常失败

这一步只为测试可控性服务，不改变运行时主逻辑。

### 17.3 自动化验证内容

扩展了：

- `test_go2_x5_getup_request`

新增两类验证：

1. `RlLocomotionEnterStaysInRlWhenInitSucceeds`

   - `RLLocomotion::Enter()` 被调用
   - `InitRL("go2_x5/robot_lab")` 成功

   - `rl_init_done == true`

经过第五轮实验，可以确认：

1. `GetUp -> RL` 的请求消费链路是稳定的
2. `RLLocomotion` 入口成功时，状态可以稳定保留在 RL
3. `RLLocomotion` 入口失败时，状态可以稳定回退到 `Passive`

第五轮实验支持如下更完整的结论：

这意味着“按下 `1` 进入稳定 RL”这一目标，在当前实验边界内已经完成闭环验证。
