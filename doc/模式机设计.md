# Go2-X5 模式机设计

## 1. 设计目标

模式机的职责是让系统在 dog-only PPO 新范式下具备可预测的运行行为和故障接管能力。

它必须解决的问题：

- 什么时候允许进入 `RL_DOG_ONLY_ACTIVE`
- arm 失败时如何降级
- body 失败时如何停机
- 策略健康异常时如何退出运行
- 什么故障必须锁存


## 2. 状态定义

建议使用以下模式：

- `BOOT`
- `PROBE`
- `PASSIVE`
- `READY`
- `RL_DOG_ONLY_ACTIVE`
- `MANUAL_ARM`
- `DEGRADED_ARM`
- `DEGRADED_BODY`
- `SOFT_STOP`
- `FAULT_LATCHED`


## 3. 状态说明

### 3.1 `BOOT`

含义：

- 进程刚启动
- 配置未完成校验
- 硬件链路未完全建立

输出策略：

- 不允许任何主动控制输出

### 3.2 `PROBE`

含义：

- 正在检查 DDS、CAN、SDK、mapping、限位、话题/IPC、初始状态

输出策略：

- 不允许进入运动控制
- 允许必要的只读探测和软初始化

### 3.3 `PASSIVE`

含义：

- 硬件在线，但系统尚未进入可执行状态

输出策略：

- body 不执行 locomotion
- arm 进入 damping / passive / hold

### 3.4 `READY`

含义：

- 系统健康、状态新鲜、可切入运行

输出策略：

- 可以等待 operator enable
- 允许 arm 小范围非 locomotion 动作测试

当前最小键盘部署语义：

- `0`: 进入 get-up 过程
- `1`: 在已完成 get-up 的前提下请求进入 `RL_DOG_ONLY_ACTIVE`
- `2`: 进入 `MANUAL_ARM` 并发送 preset arm target
- `3`: 进入 `MANUAL_ARM` 并发送 home / hold arm target
- `Space`: 只把 body 速度指令清零，不等价于 e-stop
- `Esc`: 进入 `FAULT_LATCHED`
- `R`: fault reset

### 3.5 `RL_DOG_ONLY_ACTIVE`

含义：

- PPO 控制腿
- arm 由外部链路控制
- coordinator 同步二者

输出策略：

- body 输出由 PPO + coordinator 生成
- arm 输出由外部目标 + coordinator 生成

### 3.6 `MANUAL_ARM`

含义：

- arm 被人工或上层外部控制
- 腿不执行正式 PPO locomotion

输出策略：

- body 保持安全姿态
- arm 接收有限制的外部目标

当前实现约束：

- `MANUAL_ARM` 是保留的 canonical supervisor mode，不是废弃状态
- 当前真机最小操作中，`Key[2]` / `Key[3]` 会驱动该模式
- `MANUAL_ARM` 下 body 不进入 PPO locomotion
- `MANUAL_ARM` 下 arm 允许受限 passthrough 到 preset / home 目标

### 3.7 `DEGRADED_ARM`

含义：

- arm backend 异常
- arm state stale
- arm tracking error 持续超阈值

输出策略：

- arm 切 hold / damping
- body 停止 whole-body 扰动验证
- 可以选择站立或极低速安全步态

### 3.8 `DEGRADED_BODY`

含义：

- body state stale
- Unitree DDS 写失败
- 核心稳定性约束失效

输出策略：

- 进入 soft-stop 准备
- 不允许继续 locomotion

### 3.9 `SOFT_STOP`

含义：

- 系统正在有序退出主动控制

输出策略：

- body 回默认安全姿态
- arm 回 hold / shutdown pose

### 3.10 `FAULT_LATCHED`

含义：

- 存在不可忽略故障，需要人工确认才能恢复

输出策略：

- 锁死主动控制输出
- 只允许 reset / recover 指令


## 4. 事件定义

推荐 supervisor 统一消费以下事件：

- `evt_boot_complete`
- `evt_probe_pass`
- `evt_probe_fail`
- `evt_operator_enable`
- `evt_operator_disable`
- `evt_policy_health_ok`
- `evt_policy_health_bad`
- `evt_body_state_stale`
- `evt_arm_state_stale`
- `evt_arm_tracking_error_high`
- `evt_estop`
- `evt_soft_stop_request`
- `evt_fault_reset`


## 5. 状态转移规则

### 5.1 启动路径

- `BOOT -> PROBE`
  条件：配置装载完成

- `PROBE -> PASSIVE`
  条件：probe 通过

- `PROBE -> FAULT_LATCHED`
  条件：probe 失败

- `PASSIVE -> READY`
  条件：body/arm state 连续 fresh 满足窗口要求

### 5.2 激活路径

- `READY -> RL_DOG_ONLY_ACTIVE`
  条件：
  - operator enable
  - policy health 正常
  - body state fresh
  - arm state fresh

- `READY -> MANUAL_ARM`
  条件：
  - operator 只使能 arm manual

### 5.3 运行中降级

- `RL_DOG_ONLY_ACTIVE -> DEGRADED_ARM`
  条件：
  - arm stale
  - arm backend invalid
  - arm tracking error 连续超阈值

- `RL_DOG_ONLY_ACTIVE -> DEGRADED_BODY`
  条件：
  - body stale
  - dds write failure
  - stability 关键约束触发

- `RL_DOG_ONLY_ACTIVE -> SOFT_STOP`
  条件：
  - operator disable
  - explicit soft stop request

- `RL_DOG_ONLY_ACTIVE -> FAULT_LATCHED`
  条件：
  - e-stop
  - 严重故障

### 5.4 降级恢复

- `DEGRADED_ARM -> READY`
  条件：
  - arm 恢复正常
  - 人工确认允许恢复

- `DEGRADED_BODY -> READY`
  条件：
  - body 链路恢复
  - 人工确认允许恢复

- `DEGRADED_ARM -> SOFT_STOP`
  条件：
  - 降级持续超时

- `DEGRADED_BODY -> SOFT_STOP`
  条件：
  - 进入受控退出

### 5.5 锁存恢复

- `FAULT_LATCHED -> PASSIVE`
  条件：
  - fault reset
  - 重新 probe 通过


## 6. 每个状态下的输出规则

### `PASSIVE`

- body command 输出全零或安全保持
- arm command 输出 shutdown/hold/damping

### `READY`

- body command 不允许 locomotion
- arm command 允许有限手动测试

### `RL_DOG_ONLY_ACTIVE`

- body command 由 `DogPolicyCommandFrame` 解码而来
- arm command 由外部 arm target 生成
- 允许 drift / tracking diagnostics 持续采样

### `DEGRADED_ARM`

- arm command 强制切 hold/damping
- body command 切回安全站立模板

### `DEGRADED_BODY`

- 立即准备 soft-stop
- arm 不再允许继续动态验证动作

### `SOFT_STOP`

- 以固定时间窗平滑退出
- 完成后进入 `PASSIVE`

### `FAULT_LATCHED`

- 全部主动输出冻结
- 不允许自动恢复


## 7. 故障码建议

建议统一 reason code：

- `1001`: probe failed
- `1002`: manifest mismatch
- `1003`: body state stale
- `1004`: arm state stale
- `1005`: arm tracking error too high
- `1006`: policy timeout
- `1007`: dds write fail
- `1008`: e-stop
- `1009`: command expired
- `1010`: illegal joint mapping


## 8. 模式切换日志要求

每次模式切换必须输出 `ModeEventFrame`，至少包含：

- `from_mode`
- `to_mode`
- `reason_code`
- `trigger_seq`
- `source_monotonic_ns`

要求：

- 所有模式切换可回溯
- 所有锁存故障可回放最近事件链


## 9. watchdog 与门槛建议

建议初版阈值如下（已在代码中实现）：

- body state stale: `50ms` (`Config::body_state_stale_us`)
- arm state stale: `50ms` (`Config::arm_state_stale_us`)
- policy stale: `100ms` (`Config::policy_stale_us`)
- arm tracking error 持续超限窗口: `200ms` (`Config::arm_tracking_error_window_us`)
- degraded 超时进入 soft-stop: `1.5s` (`Config::degraded_timeout_us`)
- soft_stop 持续时间: `500ms` (`Config::soft_stop_duration_us`)

说明：

- 阈值通过 supervisor Config 配置
- 可从 deploy manifest 中的 supervisor 部分读取


## 10. 实施建议

建议先在 supervisor 内部实现最小模式机，再逐步扩展事件源。

第一版最低要求：

- 启动链可跑通
- stale 能驱动降级
- e-stop 能直接锁存
- soft-stop 能受控退出

只有当这四点稳定后，才继续扩展更细粒度的恢复逻辑。
