# Go2-X5 新范式实施路线图

## 1. 目标

本路线图用于把 dog-only PPO 新范式从当前代码形态落地到可部署框架。

目标不是一次性大重写，而是按风险优先级拆成可实施阶段，保证：

- 每个阶段都能独立验收
- 每个阶段都能降低部署风险
- 不把系统拖入长期不可运行的中间态


## 2. 分阶段实施总览

建议拆为 6 个工作包。

- `WP1`: 统一部署配置
- `WP2`: 统一消息协议
- `WP3`: 引入 supervisor 与模式机
- `WP4`: 引入 hybrid coordinator
- `WP5`: 收敛 arm adapter / body adapter
- `WP6`: 建立诊断与验证闭环


## 3. WP1 统一部署配置

### 3.1 目标

把当前散落在以下位置的参数收敛为单一 deploy manifest：

- `policy/go2_x5/base.yaml`
- `policy/go2_x5/robot_lab/config.yaml`
- `launch/go2_x5_real_dual.launch.py`
- `scripts/arx_x5_bridge.py`
- 运行时 CLI 参数

### 3.2 输出物

新增：

- `deploy/go2_x5_real.yaml`
- `src/rl_sar/library/core/config/deploy_manifest_loader.*`

### 3.3 重点字段

- 频率合同
- joint mapping
- topic / IPC / DDS 配置
- timeout
- arm tracking 参数
- mode machine 阈值
- policy metadata

### 3.4 完成标准

- 所有运行模块都从同一个 manifest 读配置
- 启动日志输出 manifest hash
- 不再存在多处互相冲突的默认值


## 4. WP2 统一消息协议

### 4.1 目标

替换现有 arm 侧的：

- `Float32MultiArray`
- 裸 UDP payload

改为统一 typed frame。

### 4.2 输出物

新增：

- `src/rl_sar/include/rl_sar/protocol/go2_x5_protocol.hpp`
- `src/rl_sar/include/rl_sar/protocol/go2_x5_protocol_types.hpp`
- `src/rl_sar/src/rl_sar/protocol/go2_x5_protocol.cpp`

建议同步新增：

- `src/rl_sar/msg/*.msg`

### 4.3 优先替换链路

先替换：

- `rl_real_go2_x5 <-> arm bridge`

后替换：

- ROS2 镜像面

### 4.4 完成标准

- arm command / state 不再依赖 `Float32MultiArray`
- 所有 frame 有 `seq`
- 所有 frame 有源时间戳
- 所有 command 有 `command_expire_ns`


## 5. WP3 引入 supervisor 与模式机

### 5.1 目标

建立唯一的状态机拥有者，收拢故障接管逻辑。

### 5.2 输出物

新增：

- `src/rl_sar/include/rl_sar/runtime/supervisor/*`
- `src/rl_sar/src/rl_sar/runtime/supervisor/*`

### 5.3 先做最小模式集

第一轮冻结要求先定义完整 canonical 模式集合，避免后续各模块各自发明状态名。

第一版至少要在枚举和 reason code 中保留：

- `BOOT`
- `PROBE`
- `PASSIVE`
- `READY`
- `RL_DOG_ONLY_ACTIVE`
- `MANUAL_ARM`
- `DEGRADED_ARM`
- `DEGRADED_BODY`
- `SOFT_STOP`
- `FAULT_LATCHED`

说明：

- 第一阶段可以只优先打通 `BOOT -> PROBE -> PASSIVE -> READY -> RL_DOG_ONLY_ACTIVE` 及降级路径。
- 但 `PASSIVE` 与 `MANUAL_ARM` 的状态名不能缺席，否则会和 `模式机设计.md` 的 canonical 合同冲突。

### 5.4 完成标准

- 所有模式切换只有一个入口
- 任一 stale / timeout / backend invalid 都能驱动模式变化
- `Ctrl+C` 与 soft-stop 统一从 supervisor 发起


## 6. WP4 引入 hybrid coordinator

### 6.1 目标

把当前分散在 `rl_real_go2_x5.cpp`、state_io、output_guard、bridge 脚本中的协调逻辑收敛。

### 6.2 输出物

新增：

- `src/rl_sar/include/rl_sar/runtime/coordinator/*`
- `src/rl_sar/src/rl_sar/runtime/coordinator/*`

### 6.3 主要职责

- 接收 `DogPolicyCommandFrame`
- 接收 `ArmCommandFrame` 上层意图
- 生成最终 `BodyCommandFrame`
- 生成最终 `ArmCommandFrame`
- 判断 arm tracking error / arm stale / drift control

### 6.4 完成标准

- PPO 只进入腿部链路
- arm 外部目标不再通过兼容性的旧 whole-body 路径绕行
- 所有限幅和过期判断集中化


## 7. WP5 adapter 收敛

### 7.1 `unitree_adapter`

目标：

- 把 DDS 细节从主逻辑里隔离
- 不允许上层直接操作 `LowCmd_`

建议代码落点：

- `src/rl_sar/src/rl_sar/adapters/unitree/*`

### 7.2 `arm_adapter`

目标：

- 以 `arx5-sdk` 为主执行层
- 保持 `500Hz` 内部 servo
- 不再让 bridge timer 充当唯一 arm 控制环

建议代码落点：

- `src/rl_sar/src/rl_sar/adapters/arx/*`

### 7.3 完成标准

- body/arm 两条链路都只暴露 typed state/command frame
- 上层不再感知 DDS/CAN/SDK 细节


## 8. WP6 诊断与验证闭环

### 8.1 目标

建立可以驱动实机部署决策的指标系统。

### 8.2 输出物

新增：

- diagnostics publisher
- event logger
- drift metrics recorder
- tracking error recorder

### 8.3 核心指标

- `arm_tracking_error_norm`
- `xy_drift_error`
- `yaw_drift_error`
- `body_state_age_us`
- `arm_state_age_us`
- `policy_latency_us`
- `coordinator_jitter_us`

### 8.4 完成标准

- 所有关键指标可录包
- 所有 fault 有事件链
- 可用 `Checklist` 直接验收


## 9. 推荐代码改造顺序

建议严格按以下顺序修改代码：

1. 配置层
2. 协议层
3. supervisor
4. coordinator
5. adapters
6. ROS mirror
7. 验证脚本与日志

理由：

- 如果先改业务逻辑再改协议，容易出现中间态冲突
- 先收拢配置和协议，后面的模块才能稳定依赖


## 10. 每阶段验收门槛

### WP1 验收

- 所有模块读取统一 manifest
- 启动时能输出配置哈希

### WP2 验收

- arm 实控链路 typed frame 可通
- `seq`、时间戳、expire 都生效

### WP3 验收

- stale/fault 能正确进模式机
- `FAULT_LATCHED` 需要人工 reset

### WP4 验收

- PPO 与 arm 外部控制正式解耦
- coordinator 可输出最终 body/arm command

### WP5 验收

- adapter 能独立单测
- arm 可保持 500Hz 内部执行

### WP6 验收

- drift、tracking、freshness 可量化
- 上机 checklist 可逐项打钩


## 11. 推荐实施节奏

如果按单人主导实施，建议节奏如下：

- 第 1 周：WP1 + WP2
- 第 2 周：WP3
- 第 3 周：WP4 + WP5
- 第 4 周：WP6 + bench validation

如果多人并行：

- 一人负责配置/协议
- 一人负责 supervisor/coordinator
- 一人负责 adapters/ops


## 12. 最终目标

路线图的最终目标是把系统落成以下形态：

- `dog-only PPO runtime`
- `hybrid motion coordinator`
- `unitree adapter`
- `arm adapter`
- `deploy supervisor`
- `ops bridge`
- `single deploy manifest`
- `typed protocol`
- `measurable validation pipeline`

只有达到这个形态，新范式才算真正落地，而不是停留在训练侧定义。
