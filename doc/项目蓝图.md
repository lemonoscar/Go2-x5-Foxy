# Go2-X5 实机部署项目蓝图

## 1. 背景变更

本蓝图基于当前冻结契约和 `Go2-X5 sim2real` 新前提重新设计。

当前策略范式已经不是旧的 whole-body joint-level PPO，而是更明确的控制分工：

- PPO 只输出 `12` 个狗腿关节动作
- `6` 个机械臂关节由外部命令链路控制
- `1` 个 gripper 通道暂时保留为接口占位

这意味着整套实机系统的目标不再是“让一个 PPO 同时负责走路、稳态和机械臂跟踪”，而是：

- 用 PPO 负责腿部 locomotion 和 base stability
- 用外部 arm pipeline 负责机械臂目标生成与跟踪
- 让底盘在 arm motion 干扰下保持稳定和低漂移

这个新范式天然比旧的 whole-body PPO 更适合 sim2real，因为它降低了策略头的职责耦合，也更符合真实硬件的控制边界。


## 2. 新范式的核心原则

### 2.1 控制分工固定

实机部署时必须保持以下控制边界：

- PPO 只能控制腿
- arm 不重新塞回 PPO action head
- gripper 不并入 locomotion PPO

这是整个架构设计的第一原则，不允许在部署层重新混合。

### 2.2 协调而不混控

系统仍然要“协调”腿和臂，但不再让它们由同一策略头“混控”。

协调的含义是：

- PPO 可以看到 arm state 和 arm command
- PPO 学会在 arm motion 干扰下保持 base 稳定
- coordinator 负责把腿部 PPO 输出与 arm 外部命令并行调度

不是：

- PPO 直接输出 arm joint targets

### 2.3 sim2real 的重点从“动作统一”转成“条件稳定”

新范式下，sim2real 关注点变为：

- 机械臂动作时，底盘是否稳定
- 零速命令时，底盘是否发生 `xy drift` 或 `yaw drift`
- 低速行走时，底盘能否在 arm 扰动下保持 tracking


## 3. 系统目标

系统级目标应重新定义为：

1. 腿部策略在 arm motion 条件下保持稳态和 locomotion 质量
2. arm 外部控制链路高可靠、可跟踪、可观测
3. 系统能够严格区分“腿部策略问题”和“机械臂执行问题”
4. 即使 arm 侧失败，也能优雅退化而不是拖垮整机


## 4. 推荐模块结构

新范式下，不再需要一个“whole-body PPO 协调器”去解释 arm action，而需要一个“混合控制协调器”。

建议系统拆分为 6 个模块。

### 4.1 `deploy_supervisor`

职责：

- 启动编排
- preflight 检查
- 模式机驱动
- 故障接管
- watchdog 和心跳管理

输入：

- `BodyStateFrame`
- `ArmStateFrame`
- `PolicyHealthFrame`
- `OperatorCommandFrame`
- 子进程诊断信息

输出：

- 模式切换命令
- 接管命令
- `SOFT_STOP` / `FAULT_LATCHED` 决策

### 4.2 `locomotion_policy_runtime`

职责：

- 接收 dog-only PPO observation
- 运行策略推理
- 仅输出 `12` 维腿部动作

输入：

- base state
- leg joint state
- arm joint state
- arm command context
- height scan
- previous actions

输出：

- `DogPolicyCommandFrame`

关键约束：

- 严禁输出 arm actions
- 输出只对应 12 个腿关节

### 4.3 `hybrid_motion_coordinator`

职责：

- 收集 body state、arm state、operator command、dog PPO command
- 生成两条并行控制流：
  - `BodyCommandFrame`
  - `ArmCommandFrame`

它是“协调器”，但不是 whole-body PPO 聚合器。

职责边界：

- 负责腿部 action 解码、限幅、映射
- 负责 arm 外部目标透传、限幅、过期控制
- 负责零速锚定策略、漂移抑制模式、模式门控

### 4.4 `unitree_adapter`

职责：

- 统一管理 Unitree DDS
- 读 `rt/lowstate`
- 写 `rt/lowcmd`
- 输出 `BodyStateFrame`
- 接收 `BodyCommandFrame`

### 4.5 `arm_adapter`

职责：

- 统一管理 `arx5-sdk`
- arm 内部 servo 保持 `500Hz`
- 接收 `ArmCommandFrame`
- 输出 `ArmStateFrame`

关键要求：

- 这里是“arm tracking 层”
- 不负责底盘稳定，不运行 PPO

### 4.6 `ops_bridge`

职责：

- ROS2 镜像
- bag 记录
- 遥操作输入接入
- 可视化和诊断面板


## 5. 控制链路设计

### 5.1 腿部链路

```text
BodyState + ArmContext + CommandContext
    -> locomotion_policy_runtime
    -> 12-dim dog action
    -> hybrid_motion_coordinator
    -> BodyCommandFrame
    -> unitree_adapter
    -> rt/lowcmd
```

### 5.2 机械臂链路

```text
External arm target / arm command topic / planner
    -> hybrid_motion_coordinator
    -> ArmCommandFrame
    -> arm_adapter
    -> arx5-sdk
    -> real arm tracking
```

### 5.3 系统协调点

协调发生在 observation 与 safety 层，而不是 PPO action 层：

- PPO 看见 arm state 和 arm command
- PPO 不输出 arm command
- coordinator 决定 arm command 是否允许执行


## 6. 频率合同

推荐固定如下时钟：

- `locomotion_policy_runtime`: `50Hz`
- `hybrid_motion_coordinator`: `200Hz`
- `unitree_adapter command`: `200Hz`
- `arm target update`: `200Hz`
- `arx5-sdk internal servo`: `500Hz`
- `ops telemetry`: `50Hz` 或 `100Hz`

设计意图：

- PPO 是较低频策略层
- coordinator 是系统主控制面
- arm 底层保留更高频内部伺服

这一点和旧范式不同。旧范式更倾向于把 arm 也理解为 PPO 直驱对象；新范式下 arm 是高频执行器，PPO 只需学会“在该执行器动作时保持底盘稳定”。


## 7. Observation 设计原则

根据当前冻结契约，PPO 已能看到：

- full-body `joint_pos`
- full-body `joint_vel`
- `arm_joint_command`

所以新的部署设计中，不应重复添加无效状态，而应补充更有意义的条件变量。

建议显式补入：

- `arm_tracking_error = arm_command - arm_joint_pos`
- `arm_command_delta = arm_command_t - arm_command_{t-1}`

可选补入：

- 末端相对 base 的位置
- 末端相对 base 的速度

这样做的意义是：

- PPO 更容易区分“arm 正在动”与“arm 已经到位”
- PPO 更容易预测 COM 扰动的变化趋势


## 8. 安全设计

新范式下，安全策略也要重新分层。

### 8.1 腿部安全

- PPO action 只作用于 12 腿关节
- 所有腿部输出经过 mapping、limit clip、torque clip
- body state stale 时，直接退出 `RL_ACTIVE`

### 8.2 机械臂安全

- arm command 必须独立过期
- arm state 必须独立 freshness 检查
- arm tracking 失败时，不允许继续视为正常扰动条件

### 8.3 跨模块安全

如果 arm 出现故障，系统应优先进入：

- `DEGRADED_ARM`

而不是立刻整机失控。

`DEGRADED_ARM` 下：

- 臂进入 hold / damping
- 腿停止 whole-body 动态验证
- 允许回站立或极低速安全步态


## 9. 模式机重新定义

推荐状态：

- `BOOT`
- `PROBE`
- `PASSIVE`
- `READY`
- `RL_DOG_ONLY_ACTIVE`
- `MANUAL_ARM`
- `DEGRADED_ARM`
- `DEGRADED_BODY`
- `SOFT_STOP`
- `FAULT_LATCHED`

说明：

- `RL_DOG_ONLY_ACTIVE` 是新范式下的核心运行状态
- 它的含义是：
  - PPO 仅驱动腿
  - arm 由外部命令链路驱动
  - 两者由 coordinator 协调

关键转移条件：

- `READY -> RL_DOG_ONLY_ACTIVE`
  条件：
  - body fresh
  - arm fresh
  - policy health 正常
  - operator 允许

- `RL_DOG_ONLY_ACTIVE -> DEGRADED_ARM`
  条件：
  - arm stale
  - arm backend invalid
  - arm tracking error 持续超阈值

- `RL_DOG_ONLY_ACTIVE -> DEGRADED_BODY`
  条件：
  - lowstate stale
  - body 安全约束触发


## 10. 验证策略重排

当前冻结契约已经明确建议验证顺序应优先：

1. 静态 arm motion
2. 低速 locomotion under arm motion

我建议正式按下面顺序执行：

### 第一阶段：站立抗扰

- base velocity command = 0
- arm 执行小幅动作
- 验证：
  - `xy drift`
  - `yaw drift`
  - roll/pitch compensation

### 第二阶段：静态大扰动

- 零速站立
- arm 执行大幅 sweep / 极限合法位姿
- 验证：
  - 站立保持
  - 无异常平面漂移
  - 无安全保护误触发

### 第三阶段：低速行走

- 小速度前进
- 叠加中等幅度 arm motion
- 验证：
  - 速度跟踪
  - 足端打滑
  - gait 结构

### 第四阶段：恢复与退化

- arm 掉线
- arm state stale
- arm 指令过期
- 验证系统是否正确进入 `DEGRADED_ARM`


## 11. 配置体系调整

由于范式变化，deploy manifest 也应随之调整。

关键配置项：

- `policy_action_dim = 12`
- `policy_observation_dim = 260`
- `arm_control_mode = external`
- `gripper_mode = placeholder`
- `policy_rate_hz = 50`
- `coordinator_rate_hz = 200`
- `arm_target_rate_hz = 200`
- `arm_servo_rate_hz = 500`
- `enable_arm_tracking_error_obs = false`
- `enable_arm_command_delta_obs = false`

第一轮冻结说明：

- 为避免在不改训练侧模型的前提下制造观测维度冲突，当前部署轮次先冻结 `policy_observation_dim = 260`。
- `arm_tracking_error` 与 `arm_command_delta` 仍然是后续推荐增强项，但不进入本轮默认激活配置。
- 当后续进入观测扩展轮次时，需要同步升级：
  - 策略模型输入维度
  - deploy manifest 的 `observation_dim`
  - `policy_id`
  - 录包与诊断解释

不再使用“whole-body PPO action dim”相关配置作为实机控制真值。


## 12. 最终设计结论

新范式下，系统不应该继续围绕“whole-body PPO 部署”组织，而应该围绕“dog-only PPO + external arm controller + hybrid coordinator”组织。

这带来三个直接好处：

1. PPO 任务边界更清晰
2. 实机 arm 控制链路更接近真实工业式执行结构
3. sim2real 风险更集中在稳定性和漂移问题上，更容易验证和定位

因此，整个项目蓝图应从“统一动作输出”升级为“分层控制、条件稳定、强监督部署”的范式。
