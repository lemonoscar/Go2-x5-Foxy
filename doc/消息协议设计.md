# Go2-X5 消息协议设计

## 1. 新协议设计的前提

本协议基于新的 dog-only PPO 范式设计。

系统控制分工如下：

- PPO 仅输出 `12` 维腿部动作
- 机械臂命令由外部控制链路提供
- gripper 保留接口，但不纳入 locomotion PPO 主闭环

因此，协议层的核心目标不再是“传 whole-body PPO action”，而是清晰地区分：

- 腿部策略输出
- 机械臂外部目标
- 机械臂真实跟踪状态
- 系统协调与安全诊断


## 2. 协议设计目标

协议需要解决以下问题：

1. 区分 dog PPO command 与 arm external command
2. 明确 arm target、arm actual、arm tracking error
3. 为实机部署提供严格的时序、序号、过期控制
4. 为漂移、抗扰、条件稳定验证提供足够观测信息


## 3. 设计原则

### 3.1 策略输出与执行输出分离

PPO 的输出只代表“腿部策略建议”，不是最终整机执行命令。

最终硬件命令必须经过协调层生成：

- `DogPolicyCommandFrame`
- `ArmCommandFrame`
- `BodyCommandFrame`

### 3.2 arm 状态必须可验证

arm 相关消息必须同时表达：

- 目标值
- 实际值
- 来自真实 backend 还是 shadow
- tracking error
- 时延

### 3.3 所有命令必须可过期

任何命令消息都必须具备：

- `seq`
- `source_monotonic_ns`
- `command_expire_ns`

### 3.4 ROS2 不是主数据面

- 主数据面：typed IPC
- ROS2：镜像与运维


## 4. 统一帧头

建议所有 typed frame 使用统一头：

```text
FrameHeader
- magic: uint32
- version: uint16
- msg_type: uint16
- seq: uint64
- source_monotonic_ns: uint64
- publish_monotonic_ns: uint64
- source_id: uint32
- mode: uint16
- validity_flags: uint16
- fault_flags: uint32
- payload_bytes: uint32
```

字段说明：

- `seq`
  单调递增序号，用于检测乱序、丢包、重启。
- `source_monotonic_ns`
  源端采样或生成该数据的时刻。
- `publish_monotonic_ns`
  发送模块真正发出该消息的时刻。
- `mode`
  当前系统模式快照。
- `validity_flags`
  表示数据是否完整、是否来自 backend、是否 stale。
- `fault_flags`
  表示源端故障。


## 5. 消息类型

建议使用以下消息类型：

- `1`: `BodyStateFrame`
- `2`: `ArmStateFrame`
- `3`: `OperatorCommandFrame`
- `4`: `DogPolicyObservationFrame`
- `5`: `DogPolicyCommandFrame`
- `6`: `ArmCommandFrame`
- `7`: `BodyCommandFrame`
- `8`: `HybridDiagnosticFrame`
- `9`: `ModeEventFrame`


## 6. 状态类协议

### 6.1 `BodyStateFrame`

```text
BodyStateFrame
- header: FrameHeader
- imu_quat[4]: float32
- imu_gyro[3]: float32
- imu_acc[3]: float32
- leg_q[12]: float32
- leg_dq[12]: float32
- leg_tau[12]: float32
- base_lin_vel[3]: float32
- base_ang_vel[3]: float32
- projected_gravity[3]: float32
- lowstate_age_us: uint32
- dds_ok: uint8
- reserved[3]
```

用途：

- 提供腿部策略和协调层所需的 body 信息。
- 不把 arm 状态混进 body frame。

### 6.2 `ArmStateFrame`

```text
ArmStateFrame
- header: FrameHeader
- joint_count: uint16
- backend_mode: uint16
- q[6]: float32
- dq[6]: float32
- tau[6]: float32
- q_target[6]: float32
- tracking_error[6]: float32
- backend_age_us: uint32
- transport_age_us: uint32
- target_seq_applied: uint64
```

关键要求：

- `tracking_error = q_target - q_actual`
- 必须显式标记是否 `from_backend`
- 若为 shadow state，必须置位 `validity_flags.shadow_state`


## 7. 操作与策略类协议

### 7.1 `OperatorCommandFrame`

```text
OperatorCommandFrame
- header: FrameHeader
- cmd_source: uint16
- key_mask: uint16
- e_stop: uint8
- locomotion_enable: uint8
- arm_enable: uint8
- reserved0: uint8
- cmd_vel_x: float32
- cmd_vel_y: float32
- cmd_vel_yaw: float32
- arm_target_q[6]: float32
- gripper_target: float32
```

说明：

- 这是上层命令输入，不是硬件命令。
- 腿和臂的上层意图在这里统一进入系统。

### 7.2 `DogPolicyObservationFrame`

该消息主要用于调试、录包和可解释性，不一定在主控制闭环跨进程传输，但建议定义。

```text
DogPolicyObservationFrame
- header: FrameHeader
- base_lin_vel[3]: float32
- base_ang_vel[3]: float32
- projected_gravity[3]: float32
- velocity_commands[3]: float32
- full_joint_pos[18]: float32
- full_joint_vel[18]: float32
- last_actions[18]: float32
- height_scan[187]: float32
- arm_joint_command[6]: float32
- gripper_command[1]: float32
```

说明：

- 第一轮冻结中，该帧与当前 deploy manifest 对齐，冻结为 `260` 维观测合同。
- `arm_tracking_error` 与 `arm_command_delta` 仍是后续建议增强项，但不进入本轮默认帧定义。
- 若后续要正式引入这两个条件变量，必须同步升级：
  - `policy.observation_dim`
  - 模型权重与 `policy_id`
  - 对应录包/解析逻辑

### 7.3 `DogPolicyCommandFrame`

```text
DogPolicyCommandFrame
- header: FrameHeader
- policy_id_hash: uint64
- action_dim: uint16
- reserved0: uint16
- inference_latency_us: uint32
- leg_action[12]: float32
```

关键约束：

- 只允许包含 `12` 个腿动作
- 不允许携带 arm action


## 8. 执行类协议

### 8.1 `BodyCommandFrame`

```text
BodyCommandFrame
- header: FrameHeader
- joint_count: uint16
- reserved0: uint16
- command_expire_ns: uint64
- q[12]: float32
- dq[12]: float32
- kp[12]: float32
- kd[12]: float32
- tau[12]: float32
```

说明：

- 这是最终给 `unitree_adapter` 的腿部命令。
- 由 `hybrid_motion_coordinator` 根据 PPO action、限幅和当前模式生成。

### 8.2 `ArmCommandFrame`

```text
ArmCommandFrame
- header: FrameHeader
- joint_count: uint16
- interpolation_hint: uint16
- command_expire_ns: uint64
- q[6]: float32
- dq[6]: float32
- kp[6]: float32
- kd[6]: float32
- tau[6]: float32
- gripper_target: float32
```

说明：

- arm command 不来自 PPO，而来自外部 arm pipeline 或 operator target。
- coordinator 只负责合法性检查、模式门控和过期控制。


## 9. 诊断与事件协议

### 9.1 `HybridDiagnosticFrame`

```text
HybridDiagnosticFrame
- header: FrameHeader
- body_state_age_us: uint32
- arm_state_age_us: uint32
- policy_latency_us: uint32
- coordinator_jitter_us: uint32
- arm_tracking_error_norm: float32
- xy_drift_error: float32
- yaw_drift_error: float32
- clip_count: uint32
- dds_write_fail_count: uint32
- arm_backend_fail_count: uint32
- seq_gap_count: uint32
```

新范式下，诊断重点必须增加：

- `arm_tracking_error_norm`
- `xy_drift_error`
- `yaw_drift_error`

因为这正是 dog-only PPO 部署验证的关键指标。

### 9.2 `ModeEventFrame`

```text
ModeEventFrame
- header: FrameHeader
- from_mode: uint16
- to_mode: uint16
- reason_code: uint32
- trigger_seq: uint64
- detail_value: uint64
```


## 10. validity 与 fault 语义

建议定义以下常用位。

### 10.1 `validity_flags`

- `bit0`: payload valid
- `bit1`: from backend
- `bit2`: stale
- `bit3`: shadow state
- `bit4`: clipped
- `bit5`: expired
- `bit6`: partial
- `bit7`: fallback generated

### 10.2 `fault_flags`

- `bit0`: timeout
- `bit1`: sequence gap
- `bit2`: transport error
- `bit3`: backend invalid
- `bit4`: size mismatch
- `bit5`: illegal joint count
- `bit6`: safety reject
- `bit7`: mode reject


## 11. 时序与 freshness 规则

部署系统不能再只看“收到消息的时刻”。

必须同时计算：

- `source_age = now - source_monotonic_ns`
- `transport_age = now - publish_monotonic_ns`
- `seq_gap = current.seq - previous.seq`

推荐阈值：

- body state stale: `> 50ms`
- arm state stale: `> 50ms`
- dog policy stale: `> 100ms`
- arm command expire: `2~3` 个 coordinator 周期

特殊规则：

- 若 `ArmStateFrame` 的 `tracking_error` 持续超阈值，即使 state fresh，也应触发 `DEGRADED_ARM`
- 若 `xy_drift_error`、`yaw_drift_error` 在零速命令下持续增大，应触发部署告警


## 12. ROS2 镜像设计

建议 ROS2 只镜像以下 topic：

- `/go2_x5/body_state`
- `/go2_x5/arm_state`
- `/go2_x5/operator_cmd`
- `/go2_x5/dog_policy_cmd`
- `/go2_x5/body_cmd`
- `/go2_x5/arm_cmd`
- `/go2_x5/diag`
- `/go2_x5/mode_event`

约束：

- ROS2 只做镜像和运维，不做控制真值源
- 不能继续用 `Float32MultiArray` 作为长期实机协议


## 13. 与当前系统的迁移顺序

### 第一阶段

- 保留现有 `arm_bridge` 路径
- 把 `Float32MultiArray` 与裸 IPC payload 替换成 typed frame
- 明确区分 `DogPolicyCommandFrame` 与 `ArmCommandFrame`

### 第二阶段

- 在 coordinator 中正式引入 dog-only command 合同
- arm 外部命令不再走“伪 whole-body action”兼容路径

### 第三阶段

- 引入 `DogPolicyObservationFrame` 录包
- 在升级 `policy.observation_dim` 与 `policy_id` 后，显式补充 `arm_tracking_error` 和 `arm_command_delta`

### 第四阶段

- ROS2 降级为镜像面
- IPC 成为唯一控制数据面


## 14. 最终结论

在 dog-only PPO 新范式下，协议设计必须体现“腿部策略输出”和“机械臂外部执行”这两条链路的明确分离。

协议不再围绕“whole-body PPO action”组织，而应围绕以下四件事组织：

1. 腿部策略输出
2. arm 外部目标
3. arm 实际跟踪状态
4. 条件稳定与漂移诊断

这样才能让新的 sim2real-friendly 网络结构在部署层真正落地，而不是继续被旧的 whole-body 实机接口语义拖回去。
