# Go2-X5 部署配置规范

## 1. 目标

本规范用于建立单一 deploy manifest，解决当前配置分散的问题。

原则：

- 所有部署参数只在一处定义
- 运行模块统一从 manifest 读取
- launch 参数由 manifest 派生
- 启动时输出 manifest hash


## 2. 推荐文件

建议新增：

- `deploy/go2_x5_real.yaml`

该文件作为整套实机系统的唯一真值源。


## 3. 配置结构

建议按以下一级分组组织：

- `robot`
- `policy`
- `body_adapter`
- `arm_adapter`
- `coordinator`
- `supervisor`
- `ops`


## 4. 字段建议

### 4.1 `robot`

```yaml
robot:
  name: go2_x5
  leg_joint_count: 12
  arm_joint_count: 6
  gripper_enabled: false
  joint_mapping: [...]
  joint_lower_limits: [...]
  joint_upper_limits: [...]
  joint_velocity_limits: [...]
```

### 4.2 `policy`

```yaml
policy:
  mode: dog_only
  action_dim: 12
  observation_dim: 260
  policy_rate_hz: 50
  model_path: policy.pt
  enable_arm_tracking_error_obs: false
  enable_arm_command_delta_obs: false
```

第一轮冻结说明：

- 当前部署 contract 先冻结为与现有 dog-only 策略兼容的 `observation_dim=260`。
- `enable_arm_tracking_error_obs` 与 `enable_arm_command_delta_obs` 先作为保留开关，默认关闭。
- 只有当策略模型、runtime 观测拼接和 `policy_id` 同步升级后，才允许打开这两个开关。
- 一旦打开，`observation_dim` 必须同步调整，不能继续保持 `260`。

### 4.3 `body_adapter`

```yaml
body_adapter:
  interface: unitree_sdk2
  network_interface: eth0
  command_rate_hz: 200
  require_lowstate: true
  lowstate_timeout_ms: 50
```

### 4.4 `arm_adapter`

```yaml
arm_adapter:
  interface: arx5_sdk
  can_interface: can0
  arm_cmd_topic: /arx_x5/joint_cmd
  arm_state_topic: /arx_x5/joint_state
  arm_joint_command_topic: /arm_joint_pos_cmd
  arm_target_rate_hz: 200
  servo_rate_hz: 500
  background_send_recv: true
  controller_dt: 0.002
  require_live_state: true
  arm_state_timeout_ms: 50
  arm_tracking_error_limit: ...
```

### 4.5 `coordinator`

```yaml
coordinator:
  rate_hz: 200
  body_command_expire_ms: 15
  arm_command_expire_ms: 15
  zero_cmd_xy_drift_limit: ...
  zero_cmd_yaw_drift_limit: ...
  enable_drift_anchor: true
```

### 4.6 `supervisor`

```yaml
supervisor:
  probe_window_sec: 2.0
  degraded_timeout_sec: 1.0
  policy_timeout_ms: 100
  fault_latched_requires_manual_reset: true
```

### 4.7 `ops`

```yaml
ops:
  ros2_enabled: true
  ros2_mirror_only: true
  bridge_rmw_implementation: rmw_cyclonedds_cpp
  go2_rmw_implementation: rmw_fastrtps_cpp
  bag_enabled: true
  diagnostics_rate_hz: 50
```

补充冻结说明：

- `arm_adapter.arm_cmd_topic`
- `arm_adapter.arm_state_topic`
- `arm_adapter.arm_joint_command_topic`
- `ops.bridge_rmw_implementation`
- `ops.go2_rmw_implementation`

属于 P0 运行时部署键，必须进入 deploy manifest。
原因：

- 它们直接决定 bridge/runtime 主路径是否读取同一组 topic / RMW 配置
- 如果继续留在 launch 硬编码层，会形成“manifest + launch 双源默认值”
- 本轮目标是把这些键收敛成 manifest 默认真值源，launch 只保留 override 能力


## 5. 必须校验的约束

启动时必须自动检查：

- `policy.action_dim == 12`
- `robot.leg_joint_count == 12`
- `robot.arm_joint_count == 6`
- `arm_adapter.background_send_recv == true`
- `arm_adapter.servo_rate_hz >= coordinator.rate_hz`
- `body_adapter.command_rate_hz == coordinator.rate_hz`
- `policy.policy_rate_hz <= coordinator.rate_hz`

如不满足，直接拒绝进入 `READY`。


## 6. manifest hash 规则

启动日志必须打印：

- manifest 绝对路径
- manifest hash
- protocol version
- policy id hash

目的：

- 防止 body/arm/coordinator 读取不同配置
- 便于实机问题回溯


## 7. 与当前代码的迁移要求

以下位置的关键部署参数最终都应迁入 manifest：

- `policy/go2_x5/base.yaml`
- `policy/go2_x5/robot_lab/config.yaml`
- `launch/go2_x5_real_dual.launch.py`
- `scripts/arx_x5_bridge.py`
- runtime CLI args

允许暂时保留旧参数路径，但必须满足：

- 旧参数只是 manifest override
- manifest 始终是默认真值源


## 8. 版本管理建议

建议 manifest 中新增：

```yaml
meta:
  manifest_version: 1
  protocol_version: 1
  policy_id: ...
  generated_at: ...
```

要求：

- 版本升级不能静默
- protocol 与 manifest 版本变更要显式记录


## 9. 最终目的

部署配置规范的作用不是“整理 YAML”，而是让系统具有以下工程属性：

- 可复现
- 可审计
- 可比较
- 可回滚

没有单一 deploy manifest，就不可能有严谨的实机部署。
